# Процесс релизов

Этот репозиторий публикует GitHub Releases из верхней секции `CHANGELOG.md`. Публичная часть релиза должна быть короткой и понятной пользователю. Вся техничка хранится отдельно в `docs/releases/`.

## Целевая схема

```text
main
  └─ release/vX.Y.Z
       ├─ vX.Y.Z-pre1
       ├─ vX.Y.Z-pre2
       └─ ...
          ↓
        merge -> main
          ↓
        vX.Y.Z
```

## Как готовить релиз

1. От актуального `main` создайте ветку `release/vX.Y.Z`.
2. Обновите `version` в [pubspec.yaml](../pubspec.yaml).
3. Добавьте в начало [CHANGELOG.md](../CHANGELOG.md) секцию `## vX.Y.Z` или `## vX.Y.Z-preN`.
4. Заполните секцию короткими пользовательскими bullet-пунктами.
5. Создайте технический файл [docs/releases/vX.Y.Z.md](releases/README.md), если нужно сохранить детали реализации, ограничения или rollout-заметки.
6. Во время обкатки публикуйте теги `vX.Y.Z-preN` прямо из release-ветки.
7. Когда release-ветка проверена, смержите её в `main`.
8. Только после merge в `main` публикуйте стабильный тег `vX.Y.Z`.

## Что должно быть в GitHub Release

Видимая пользователю часть:

- 2–5 коротких bullet-пунктов;
- понятный язык без внутренних терминов, если они не нужны пользователю;
- только итог того, что реально поменялось.

В отдельном техническом файле `docs/releases/vX.Y.Z.md`:

- детали реализации;
- ограничения и caveats;
- миграционные заметки;
- rollout / verification notes, если они важны.

## Что делает workflow релизов

- валидирует текст секции под нужный тег в `CHANGELOG.md`;
- собирает release body только из этой короткой пользовательской секции;
- автоматически добавляет ссылку на подробный technical notes файл, если он существует;
- собирает и прикладывает артефакты и `*.sha256`;
- публикует stable-теги как обычные релизы, а `-preN` — как GitHub prerelease;
- оставляет Linux RPM только для stable-тегов, потому что RPM не принимает prerelease-маркеры в поле версии.

## Правила текста релиза

Нужно:

- писать через пользовательский результат;
- держать формулировки короткими;
- выносить техничку из публичного списка изменений.

Нельзя:

- писать TODO-пункты;
- упоминать CI, workflow, checksum, тестовые команды и внутренние процессы в пользовательской части;
- смешивать краткие release notes с техническим changelog.

## Branch build для обкатки

Тяжёлая Android branch build теперь предназначена только для release-веток и ручного запуска. Это означает:

- новый коммит в обычную feature/fix/docs-ветку не должен автоматически запускать полную Android-сборку;
- `main` не используется как постоянный прогревочный стенд;
- предрелизная проверка идёт в `release/vX.Y.Z`.
