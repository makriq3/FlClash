# Процесс релизов

Этот репозиторий собирает GitHub Releases напрямую из `CHANGELOG.md`. Источник правды для публичного релиза один: верхняя секция changelog для нужного тега. Подробный технический changelog хранится отдельно в `docs/releases/`.

## Как подготовить релиз

1. Обновите `version` в [pubspec.yaml](../pubspec.yaml).
2. Добавьте новую секцию `## vX.Y.Z` в начало [CHANGELOG.md](../CHANGELOG.md).
3. Заполните секцию короткими пользовательскими bullet-пунктами без технических подробностей и служебных заметок.
4. Если нужен подробный разбор, создайте файл в [docs/releases/](releases/README.md) по шаблону технического changelog.
5. Создайте и отправьте тег `vX.Y.Z`.
6. Для предрелизов используйте тег вида `vX.Y.Z-preN`: workflow опубликует GitHub prerelease и соберёт весь стандартный набор артефактов, кроме Linux RPM.

## Что делает workflow

- собирает артефакты для поддерживаемых платформ;
- сначала валидирует формулировки секции под новый тег и останавливает релиз, если там остались todo-пункты или служебный шум;
- генерирует `release.md` из пользовательской секции `CHANGELOG.md`, без автокоммитов обратно в `main`;
- если существует `docs/releases/vX.Y.Z.md`, автоматически добавляет ссылку на подробный технический changelog в GitHub Release;
- прикладывает бинарники и `*.sha256` к GitHub Release;
- публикует stable-теги как обычные релизы, а теги с суффиксом prerelease как GitHub prerelease;
- оставляет Linux RPM только для stable-тегов, потому что формат RPM не принимает semver prerelease-маркеры в поле версии.

## Правила для текста релиза

- Пишите в прошедшем времени и через итог для пользователя, а не через внутренние TODO.
- Не добавляйте служебные пункты вроде `Обновить changelog`.
- Не смешивайте release notes с инструкциями по сопровождению, checksum-подсказками, CI и внутренними комментариями.
- Не добавляйте в пользовательский changelog `Верификацию`, названия тестовых команд, workflow и внутренние идентификаторы.
- Для технических деталей, ограничений, миграций и заметок по проверке используйте отдельный файл в `docs/releases/`.
