# Правила ведения репозитория

Этот репозиторий публичный. Любые изменения должны выглядеть аккуратно, понятно и по-продуктовому, а не как внутренний черновик.

## Базовые правила

- Основной язык публичной документации, release notes и описаний — русский.
- `README.md`, `CHANGELOG.md`, release notes и описание репозитория должны быть понятны обычному пользователю, а не только разработчику.
- В публичных текстах не допускаются TODO-пункты, внутренние пометки, куски CI-логики и служебный шум.
- Каждое заметное изменение должно обновлять релевантную документацию в этом же PR.

## Ветки

### Основные ветки

- `main` — только проверенное состояние, готовое к стабильному релизу.
- `release/vX.Y.Z` — подготовка конкретного релиза.
- `feature/...`, `fix/...`, `docs/...`, `ci/...` — рабочие ветки для отдельных задач.

### Правило релизной обкатки

Каждый новый релиз проходит через отдельную release-ветку:

1. От `main` создаётся `release/vX.Y.Z`.
2. Все правки для релиза попадают только в неё.
3. Из этой ветки публикуются one or more предрелизы `vX.Y.Z-preN`.
4. После проверки release-ветка мержится в `main`.
5. Уже из итогового состояния выпускается стабильный тег `vX.Y.Z`.

Напрямую выпускать стабильный релиз из feature/fix-ветки нельзя.

## Правила для релизов

- GitHub Release должен начинаться с короткого и понятного списка изменений для пользователя.
- Вся техничка, ограничения, rollout-заметки и детали реализации выносятся в `docs/releases/vX.Y.Z.md`.
- `CHANGELOG.md` остаётся коротким пользовательским источником правды для GitHub Release.
- Предрелизы должны явно маркироваться тегами вида `vX.Y.Z-preN`.
- Стабильный релиз нельзя публиковать, если для него не было хотя бы одной предрелизной обкатки в release-ветке.

## CI и сборки

- Тяжёлые веточные Android-сборки не должны стартовать на каждый коммит в любые ветки.
- Автоматическая branch build используется только для `release/**` и ручного запуска.
- `main` не используется как площадка для бесконечной обкатки промежуточных сборок.
- Если изменение влияет только на документацию, не надо привязывать к нему лишние тяжёлые build-процессы.

## Pull Request checklist

Перед merge убедись, что:

- обновлены `README.md` / `CHANGELOG.md` / release notes, если это нужно;
- текст релиза написан по-человечески и без техмусора;
- ветка и теги соответствуют релизной схеме;
- CI отражает реальный intended workflow, а не случайную историческую настройку.
